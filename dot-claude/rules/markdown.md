# Markdown Rules

## 行長

- 散文は 80 桁で折り返す
- table / fenced code / 長い URL は例外（折らない）

## 見出し

- ATX スタイル (`#`) を使う。Setext（下線）は使わない
- H1 はファイルに 1 つだけ
- `#` の後にスペース、見出しの前後に空行

## リスト

- 箇条書きは `-` で統一（`*` / `+` は使わない）
- 番号付きは「遅延番号（全部 `1.`）」を使う。差分が小さく並び替えしやすい
- ネストは 2 スペース

## Table

### 使ってよい場面

- 2〜4 列で、各セルが短い単語〜短文
- 列方向に同種データが並び、並列比較が目的

### 使ってはいけない場面

- 1 セルに 1 文以上の長文が入る
- **1 セルに複数アイテムを `,` や `/` 区切りで詰める**（短いアイテムでも列幅が破綻する。URL / アクション / ファイル名などが該当）
- 列が 5 以上になる
- 行ごとにセルの長さがバラバラ（不均一分布）
- 50 行を超える（縦長すぎて走査しづらい）

### 代替パターン

- 長文を含むなら「小見出し + 段落」か「定義リスト風の箇条書き」にする
- 例: `- **ファイル名**: 種別 / サイズ / 更新日 — 1 行要約`
- 1 セルに複数アイテムを詰めている場合は **1 行 1 アイテム** に分割する（他の列の値は重複しても可）
- 50 行を超える大規模 table は、カテゴリ別に **サブ見出しで分割** する
- 長い補足（環境変数リスト、enum 値一覧等）は表外に `> (※1) ...` の **脚注** として出す
- どうしても広い table が必要なら TSV/CSV に外出ししてリンクする

### Table を書くときの形式

- table の前後に必ず空行を入れる（parser の false positive 回避）
- pipe の前後にスペース
- 列幅はその列の最長セルに揃える（diff を安定させる）

## コードブロック

- 常に fenced (```` ``` ````) を使う。indent code block は使わない
- 言語指定を必ず書く（`ts` / `bash` / `json` など）

## リンク

- `[text](url)` のインラインリンクを基本とする
- 同じ URL を複数箇所で使うなら reference link にする
- URL をそのまま裸で貼らない（意味のある text を付ける）

## GFM Alerts

- `> [!NOTE]` / `[!IMPORTANT]` / `[!WARNING]` を使う
- 1 ドキュメントあたり 1〜2 個まで。連続配置しない

## 段落・改行

- 段落区切りは空行
- 強制改行（行末スペース 2 個）は原則使わない
